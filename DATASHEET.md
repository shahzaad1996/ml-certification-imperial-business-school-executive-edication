# Datasheet: BBO Capstone Project Dataset

> Framework: Gebru et al. (2021) "Datasheets for Datasets"

---

## 1. Motivation

**Why was this dataset created?**  
This dataset was created to support a Black-Box Optimisation (BBO) capstone project as part of the Imperial Business School Executive Education Machine Learning programme. The goal is to find the input combination that maximises each of eight unknown objective functions using a limited query budget. Each function represents a realistic real-world optimisation scenario (see Composition below), and queries are expensive — meaning the dataset must be built strategically rather than exhaustively.

**What task does it support?**  
Sequential, query-efficient black-box optimisation. Specifically: fitting surrogate models (Gaussian Processes, SVR ensembles, MLP ensembles) to observed input–output pairs and using acquisition functions (UCB, predictive entropy) to select the next query point most likely to improve the best-observed output.

**Who created it and on whose behalf?**  
Created by the student as part of an individual capstone submission. No external funding or institutional data-collection infrastructure was involved.

---

## 2. Composition

**What does the dataset contain?**  
The dataset consists of eight independent input–output tables, one per objective function. Each row represents a single black-box evaluation: a vector of normalised continuous inputs `x ∈ [0, 1]^d` and the corresponding scalar output `y`.

| Function | Domain scenario | Input dim `d` | Initial observations | Total observations (incl. weeks 13–24) | Output range (all rounds) |
|---|---|---|---|---|---|
| F1 | 2D radiation field — detect contamination source | 2 | 10 | 22 | [−0.0077, ~0] |
| F2 | 2D noisy ML model log-likelihood | 2 | 10 | 22 | [−0.066, 0.697] |
| F3 | 3D drug compound side-effect minimisation | 3 | 15 | 27 | [−0.508, −0.035] |
| F4 | 4D warehouse hyperparameter tuning | 4 | 30 | 42 | [−33.24, 0.303] |
| F5 | 4D chemical process yield optimisation | 4 | 20 | 32 | [103.5, 6117.3] |
| F6 | 5D cake recipe optimisation | 5 | 20 | 32 | [−2.747, −0.337] |
| F7 | 6D ML hyperparameter tuning | 6 | 30 | 42 | [0.003, 2.908] |
| F8 | 8D high-dimensional black-box optimisation | 8 | 40 | 52 | [5.59, 9.895] |

**Format:**  
- Inputs: NumPy `.npy` arrays, shape `(n_observations, d)`, dtype `float64`.  
- Outputs: NumPy `.npy` arrays, shape `(n_observations,)`, dtype `float64`.  
- Notebooks: one Jupyter notebook per function (`project_f1.ipynb` – `project_f8.ipynb`).

**Are there any gaps or missing values?**  
No missing values. Four query outputs for F1 (weeks 19–22) returned effectively 0 (wk19 and wk20 returned exactly 0; wk21 returned 3.38e-96; wk22 returned exactly 0), indicating the queried locations were outside the non-zero support of the function. The wk23 probe at `[0.495, 0.404]` returned −8.17e-5, the first non-trivial F1 signal since wk15; the wk24 probe at `[0.566, 1.0]` returned exactly 0 again. The wk24 F3 query at `[0.398, 0.013, 0.995]` returned −0.508 (worst student result for F3), extending the observed output range. These are valid observations, not absent data.

**Is the dataset self-contained?**  
Yes. All `.npy` files and notebooks are stored in the repository under `Capstone_project_stage_2/`. No external database connection is required.

## 3. Collection Process

**How were queries generated?**  
Queries were generated in two phases:

1. **Initial data (provided):** The initial inputs and outputs for each function were provided as `.npy` files at the start of the project. These represent a mix of space-filling and possibly random initial evaluations by the course organisers.

2. **Student-generated queries (weeks 13–24):** Twelve additional query points per function were generated by the student over twelve iterative submissions. The strategy evolved as follows:

| Submission round | Primary strategy |
|---|---|
| 1 | Random sampling + GP-UCB initialisation |
| 2–3 | GP-UCB with high β (exploration), RBF + WhiteKernel |
| 4 | GP + SVR bootstrap ensemble (25 models, RBF, C=10) added |
| 5–6 | Multi-surrogate comparison: GP + SVR + MLP ensemble |
| 7–8 | MLP bootstrap ensemble (50 models, hidden=(64,64)) as primary for high-dim; classification acquisition (LR + SVC + MLP, predictive entropy) for signal-sparse functions |
| 9 | Consensus across GP-UCB + SVR-UCB + MLP-UCB; per-function β/κ schedule |
| 10 | Clustering-informed refinement: queries target identified cluster centroids and extend trajectories; focus radius sized by cluster spread |
| 11 | Boundary probing and validation: systematic edge-of-region probes, increased κ on key functions (F4: κ=4, F5: κ=8), reproducibility re-query on F7 |
| 12 | Exploitation refinement and extremes testing: targeted exploitation near known optima (F7 new best), extreme-parameter probes (F3, F5, F6, F8) to map boundary conditions |

**Acquisition parameters used:**  
- Low-dim (F1, 2D): UCB with β = 6.0 on a 100×100 grid; also entropy-based classification acquisition.  
- Low-dim (F2, 2D): UCB with κ = 3 on a 100×100 grid.  
- Mid-dim (F3, 3D): GP-UCB with κ = 30 on 5,000 random candidates.  
- Mid-dim (F4, 4D): GP-UCB with κ = 4 on 5,000 random candidates.  
- Mid-dim (F5, 4D): GP-UCB with κ = 8 on 5,000 random candidates.  
- Mid-dim (F6, 5D): GP-UCB with κ = 4 on 20,000 random candidates.  
- High-dim (F7, 6D): GP-UCB with κ = 3 on 20,000 random candidates.  
- High-dim (F8, 8D): GP-UCB with κ = 10.0 on 50,000 random candidates.

**Time frame:**  
Queries were submitted over approximately twelve weekly rounds (weeks 13–24 of the programme calendar).

**Were any queries rejected or discarded?**  
No. All submitted queries and their returned outputs were appended to the dataset, including zero-output observations for F1 (wk19–22) and the deliberate re-query of the wk22 point for F7 at wk23 (which confirmed deterministic function behaviour).

## 4. Preprocessing and Uses

**Have any transformations been applied?**  

| Transformation | Applied to | Purpose |
|---|---|---|
| Log-sign transform: `sign(y) · log1p(|y|)` | F1 outputs | Compresses the extreme range of near-zero and non-zero values for GP fitting |
| No normalisation of inputs | All functions | Inputs are already in `[0, 1]` by design |
| Binary labelling at 75th-percentile threshold | F1 (classification layer) | Converts regression outputs to class labels for entropy-based acquisition |
| Bootstrap resampling | SVR and MLP ensemble training | Produces empirical uncertainty estimates in lieu of analytical confidence intervals |

**What are the intended uses?**  
- Training and evaluating surrogate models (GP, SVR, MLP) for sequential BBO.  
- Benchmarking acquisition strategies (UCB, Expected Improvement, predictive entropy).  
- Academic study of exploration–exploitation trade-offs across varying input dimensionalities.

**What uses should be avoided?**  
- **Causal inference**: the data does not record the true generative mechanism; input–output relationships cannot be interpreted causally.  
- **High-stakes deployment**: the functions are pedagogical proxies, not validated real-world simulators. Results should not be used to make real chemical, radiation, or ML deployment decisions without independent validation.  
- **Training general-purpose models**: the datasets contain at most 52 observations — far too few for training any model beyond a lightweight surrogate.

## 5. Distribution and Maintenance

**Where is the dataset available?**  
The dataset (`.npy` files and notebooks) is available in the public GitHub repository for this project. See the README for the repository link.

**Terms of use:**  
The initial `.npy` data files were provided by Imperial Business School as part of the capstone challenge. Usage is restricted to the capstone project. The student-generated query points and all code are available under the MIT licence (see repository).

**Who maintains it?**  
The student author maintains the dataset for the duration of the programme. No ongoing maintenance is planned after the final submission. If the initial data files are updated or corrected by the course organisers, the student-generated queries and notebooks would need to be re-run from the updated baseline.

**Will the dataset be updated?**  
The student-generated queries will not be updated after the twelfth submission. The initial data files are static as provided by the course.

**Are there any known issues or biases?**  
- The initial query locations for each function were generated by the course and are not fully disclosed. Their spatial distribution may favour certain regions of the input space.  
- For F1, outputs are essentially zero across most of the space, making the dataset highly imbalanced for classification tasks.  
- For F5, exploratory queries at low parameter values (wk18, wk20) returned outputs 20–60× lower than the peak, creating outliers that can distort surrogate fit.  
- For F4, outputs span from −33.24 to +0.303, with extreme negative outliers from boundary exploration (wk14–wk16) that dominate the GP's learned length scales.  
- The dataset is small (22–52 observations per function), which means surrogate models are sensitive to outliers and bootstrap uncertainty estimates have high variance.
